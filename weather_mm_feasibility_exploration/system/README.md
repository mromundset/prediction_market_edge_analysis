# Weather MM validation system

Read-only, zero-capital infrastructure to measure the **captured** market-making edge `s`
on Kalshi weather ladders (see [../IMPLEMENTATION_PLAN.md](../IMPLEMENTATION_PLAN.md)).
The same `kalshi_client` is reused across phases.

## Components

| file | phase | what it does |
|---|---|---|
| `kalshi_client.py` | all | read-only Kalshi market-data client (correct `orderbook_fp` parsing, trade dedup). No auth. |
| `recorder.py` | **0** | continuous book + trade tape recorder → `tape/*.jsonl` |
| `simulator.py` | **1** | replays the tape, rests simulated quotes, models fills/queue, marks to resolution → H1/H2 |

## Phase 0 — run the recorder

```bash
cd weather_mm_feasibility_exploration/system
python recorder.py
# config via env:
#   TAPE_DIR=./tape           output dir
#   BOOK_CADENCE_S=20         full-book snapshot interval
#   TRADE_CADENCE_S=10        trade poll interval
#   ACTIVE_WINDOW_H=30        only record markets within this many hours of close
```

It records the 5 liquid cities (NYC/CHI/MIA/LAX/AUS), only while a market is within
`ACTIVE_WINDOW_H` of close (the tradeable window; the >24h zone is the documented
loss zone and is skipped). It is crash-safe (trade-id dedup is rebuilt from the day's
file on restart), rotates files by UTC day, and shuts down cleanly on Ctrl-C.

**It must run continuously** (an always-on box) to build a usable multi-day tape; that is
the substrate the historical study lacked and that the Phase-1 simulator needs.

### Output (append-only JSONL, rotated by UTC day)
- `books_YYYYMMDD.jsonl` — `{"ty":"book","t","tk","yb","ya","yes":[[px,sz]],"no":[[px,sz]]}`
- `trades_YYYYMMDD.jsonl` — `{"ty":"trade","t","tk","p","c","taker_yes":0/1,"id"}`
- `meta.jsonl` — `{"ty":"meta","tk","floor","cap","close","result"}` (written after settlement)

`tape/` is git-ignored (data, regenerated by running the recorder).

## Phase 1 — run the simulator

```bash
cd weather_mm_feasibility_exploration/system
python simulator.py             # runs the self-test, then replays tape/
python simulator.py --selftest  # deterministic correctness test only
```

Replays the tape, rests simulated quotes, and reports per (mode × queue assumption):
- **H1** fills + contracts captured,
- **H2** held-to-resolution PnL/contract on *our* fills (the decisive number),
- short-horizon adverse selection (mid move right after our fill).

**Modes:** `JOIN` (quote at the touch, compete on queue) vs `IMPROVE` (quote 1 tick
inside, jump the queue). **Queue:** `front` (Q_ahead=0) / `realistic` (behind the touch
size) / `pessimistic` (×1.5). The fill model invents no fills — a resting bid only fills
from real SELL-YES trades that overflow the queue ahead of it (ask side symmetric).

Correctness is locked by `--selftest` (deterministic synthetic tape with hand-computed
answers for queue consumption, inventory, settlement PnL, the timing rule, and JOIN/IMPROVE
eligibility).

**Early observation (100s of tape):** queue position dominates — JOIN at the back of the
queue fills ~10× less than the front; IMPROVE (jumping the queue) fills far more but earns
a thinner spread. The decisive **H2 PnL/contract** requires tape spanning full days through
settlement; run the recorder continuously first, then re-run this.
